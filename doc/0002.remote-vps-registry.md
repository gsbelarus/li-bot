# Remote VPS Management Subsystem Requirements

## Purpose

The Remote VPS Management subsystem provides operators with a single place in the Control Center application to register, inspect, update, and remove remote VPS records and to track interactions between the Control Center Server and each VPS Remote Controller.

The subsystem must support the architecture defined in [0001.architectural-overview.md](c:/Golden/Andersen/li-bot/doc/0001.architectural-overview.md):

- Each remote VPS runs a Remote Controller.
- The Control Center Server manages many Remote Controllers.
- Operators work through the Admin Dashboard.
- MongoDB Atlas stores operational data.

## Scope

This specification covers:

- Persistent storage of VPS registry records.
- Persistent storage of Control Center Server to Remote Controller interaction logs.
- Admin Dashboard UI for viewing and managing the VPS registry.
- Main application screen layout used by this subsystem.

This specification does not cover:

- Browser automation inside Chrome.
- Bot command execution logic.
- Gateway protocol details between Chrome Extension and Open Claw Gateway Server.

## Goals

- Replace informal VPS tracking with a structured registry.
- Allow operators to create, edit, and delete VPS records from the web UI.
- Give operators visibility into VPS identity, status, and last known activity.
- Preserve an audit trail of communications between the Control Center Server and each VPS Remote Controller.
- Standardize the application shell so the VPS registry fits into a broader multi-module control center UI.

## Actors

- Operator: manages VPS records and reviews logs.
- Control Center Server: stores VPS data, sends commands to Remote Controllers, and records interaction events.
- Remote Controller: accepts requests from the Control Center Server and returns responses, status, and errors.

## Functional Requirements

### VPS Registry

The system shall:

1. Allow an operator to create a VPS record.
2. Allow an operator to edit an existing VPS record.
3. Allow an operator to delete a VPS record.
4. Display the list of registered VPS entries in a dedicated working screen.
5. Support searching and filtering the VPS list.
6. Show the current operational status of each VPS based on the latest known controller communication.
7. Show the last contact time for each VPS.
8. Preserve historical interaction logs even if a VPS record is later disabled or deleted, unless explicit data retention rules require otherwise.

### Interaction Logging

The system shall:

1. Record every request sent from the Control Center Server to a Remote Controller.
2. Record every response returned by a Remote Controller to the Control Center Server.
3. Record communication failures, timeouts, retries, and transport errors.
4. Link every log entry to the corresponding VPS record.
5. Allow an operator to review interaction history for a selected VPS.
6. Store enough metadata to reconstruct what was requested, when it happened, and whether it succeeded.

### Main Application Shell

The application main screen shall:

1. Use a left-aligned vertical main menu.
2. Reserve the remaining horizontal space as the working area.
3. Use a title area at the top of the working area.
4. Use the remaining space below the title area for data display and interaction.
5. Keep this shell consistent across VPS registry list, create form, edit form, and detail/log views.

## Database Requirements

MongoDB shall contain at least two primary collections for this subsystem:

- `remote_vps`
- `remote_vps_interaction_logs`

### Collection: `remote_vps`

This collection stores the master list of managed VPS instances.

#### Required fields

- `_id`: ObjectId, primary identifier.
- `name`: string, human-readable VPS name shown in UI.
- `host`: string, DNS name or IP address used to reach the Remote Controller.
- `port`: number, Remote Controller port.
- `protocol`: string enum, expected values `http` or `https`.
- `environment`: string enum or string value, such as `production`, `staging`, `test`, `other`.
- `region`: string, optional geographic or provider region label.
- `provider`: string, VPS provider name.
- `controllerVersion`: string, last reported Remote Controller version.
- `status`: string enum.
- `statusReason`: string, short explanation for the current status.
- `lastSeenAt`: datetime, last successful communication timestamp.
- `lastHealthCheckAt`: datetime, last health check attempt timestamp.
- `lastHealthCheckResult`: string enum, such as `success`, `failed`, `timeout`, `unknown`.
- `tags`: array of strings for grouping and filtering.
- `notes`: string, operator-maintained notes.
- `isEnabled`: boolean, indicates whether the VPS is active in the control plane.
- `createdAt`: datetime.
- `updatedAt`: datetime.
- `createdBy`: string or user reference.
- `updatedBy`: string or user reference.

#### Recommended status values

- `online`: controller reachable and healthy.
- `degraded`: controller reachable but returned warnings or partial failures.
- `offline`: controller not reachable.
- `disabled`: record intentionally disabled by operator.
- `unknown`: no successful health data yet.

#### Optional future fields

- `credentialsRef`: reference to a secure secret, not raw credentials.
- `vpsOs`: string.
- `chromeVersion`: string.
- `gatewayInstalled`: boolean.
- `botInstalled`: boolean.
- `controllerCapabilities`: array of strings.

#### Validation rules

- `name` must be required and non-empty.
- The tuple of `protocol`, `host`, and `port` should be unique among active records.
- `port` must be an integer in the valid TCP range.
- `status` must be restricted to the approved enum.
- Raw secrets, passwords, and tokens must not be stored directly in this collection.

### Collection: `remote_vps_interaction_logs`

This collection stores the communication trail between the Control Center Server and Remote Controllers.

#### Required fields

- `_id`: ObjectId, primary identifier.
- `vpsId`: ObjectId, foreign key reference to `remote_vps._id`.
- `correlationId`: string, groups request, retry, and response records for a single operation.
- `direction`: string enum, such as `outbound_request`, `inbound_response`, `internal_event`.
- `interactionType`: string enum, such as `health_check`, `command_dispatch`, `status_pull`, `configuration_update`, `registration`, `manual_test`.
- `requestMethod`: string, such as `GET`, `POST`, `PATCH`.
- `requestPath`: string, target endpoint or logical command path.
- `requestPayload`: object, sanitized payload sent by the Control Center Server.
- `responseStatusCode`: number, optional for transport failures.
- `responsePayload`: object, sanitized response body.
- `result`: string enum, such as `success`, `failed`, `timeout`, `rejected`, `retrying`.
- `errorCode`: string, optional internal or transport error code.
- `errorMessage`: string, optional summarized error detail.
- `durationMs`: number, elapsed time for the interaction.
- `attempt`: number, retry attempt number starting from 1.
- `initiatedBy`: string enum, such as `system`, `operator`, `scheduler`.
- `initiatedByUserId`: string or user reference, optional when initiated by an operator.
- `createdAt`: datetime, event timestamp.

#### Logging rules

- Sensitive values must be masked before persistence.
- Large payloads may be truncated according to retention policy, but a summary must still be preserved.
- Each outbound request should create at least one log entry.
- Each corresponding response or failure should create a linked log entry using the same `correlationId`.
- Logs must be queryable by `vpsId`, `result`, `interactionType`, and time range.

#### Retention and indexing

- The system should index `vpsId` plus `createdAt` for timeline queries.
- The system should index `correlationId` for request-response grouping.
- The system should define a retention policy for old logs, for example archive or TTL after an agreed operational period.

## API-Level Requirements

The Control Center Server should expose backend endpoints sufficient for the Admin Dashboard to manage the registry and inspect logs.

### VPS registry endpoints

- `GET /api/vps`: list VPS records with paging, sorting, filtering, and search.
- `GET /api/vps/:id`: fetch one VPS record.
- `POST /api/vps`: create a VPS record.
- `PATCH /api/vps/:id`: update a VPS record.
- `DELETE /api/vps/:id`: delete a VPS record or perform soft delete based on final implementation choice.

### Interaction log endpoints

- `GET /api/vps/:id/logs`: list interaction logs for one VPS with filters.
- `GET /api/vps/:id/logs/:logId`: fetch one detailed interaction log.

### Operational endpoints

- `POST /api/vps/:id/test-connection`: manually trigger a connectivity test.
- `POST /api/vps/:id/health-check`: manually trigger a health check.

## UI Requirements

### Application Shell

The Control Center application shall use a two-pane layout:

- Left pane: fixed-width vertical main menu.
- Right pane: flexible working area.

The left main menu shall:

- Be visible on every authenticated main screen.
- Provide top-level navigation entries.
- Include at minimum a `Remote VPS` menu item for this subsystem.
- Support highlighting of the active section.

The right working area shall contain:

- A top title bar with the current screen title.
- Optional action buttons aligned within the title area.
- A content region below the title bar that fills the remaining available space.

### Remote VPS List Screen

The `Remote VPS` screen shall be the primary working screen for this subsystem.

#### Title area

The title area shall include:

- Screen title: `Remote VPS`.
- Short subtitle or helper text explaining the screen purpose.
- Primary action button: `Add VPS`.
- Optional quick actions such as `Refresh`.

#### Data area

The data area shall display a table or grid with one row per VPS.

Minimum columns:

- Name
- Host
- Port
- Environment
- Provider
- Status
- Last seen
- Controller version
- Actions

Supported actions per row:

- View details
- Edit
- Delete
- Open logs
- Test connection

Supported list capabilities:

- Text search by name, host, provider, or tags.
- Filter by status.
- Filter by environment.
- Sort by name, status, created date, or last seen time.
- Pagination or infinite scrolling.
- Empty state when there are no VPS records.

### Create VPS Screen

The create screen shall open in the working area and provide a form with at least these fields:

- Name
- Host
- Port
- Protocol
- Environment
- Region
- Provider
- Tags
- Notes
- Enabled flag

The form shall:

- Validate required fields before submit.
- Prevent duplicate active host/protocol/port combinations.
- Display field-level validation messages.
- Provide `Create` and `Cancel` actions.

### Edit VPS Screen

The edit screen shall:

- Reuse the same layout and most fields from the create screen.
- Prepopulate the form with the existing VPS record.
- Provide `Save` and `Cancel` actions.
- Warn the user before losing unsaved changes.

### Delete VPS Flow

The delete flow shall:

- Require explicit confirmation.
- Clearly show which VPS record will be deleted.
- Warn about operational consequences.
- Define whether deletion is soft delete or hard delete.
- Preserve or archive interaction logs according to retention rules.

### VPS Details and Logs View

The subsystem should provide a detail view or side panel for a selected VPS.

It should display:

- Full VPS metadata.
- Current status and last health data.
- Recent interaction logs.
- Ability to open a more detailed log history view.

The logs view should support:

- Filtering by result.
- Filtering by interaction type.
- Filtering by time range.
- Viewing request and response payload summaries.
- Viewing error information when interactions fail.

## Non-Functional Requirements

### Performance

- The VPS list screen should load the first page within an acceptable administrative UI latency target.
- Log queries should support efficient filtering by VPS and recent time ranges.

### Reliability

- Logging failures should not silently discard critical operational events.
- The system should distinguish between unreachable controller, controller error response, and Control Center internal failure.

### Security

- Only authorized users may create, edit, or delete VPS records.
- Only authorized users may view detailed interaction logs.
- Secrets must be stored in secure secret storage, not in plaintext VPS records or logs.
- Logged payloads must be sanitized before persistence and display.

### Auditability

- Every create, update, and delete action on a VPS record should be attributable to a user.
- Interaction logs must provide a clear chronological history for operational troubleshooting.

## Suggested Primary User Flows

### Register a new VPS

1. Operator opens `Remote VPS` from the left menu.
2. Operator clicks `Add VPS`.
3. Operator enters connection and metadata fields.
4. System validates the input.
5. System stores the VPS record.
6. System returns the operator to the list and shows the new entry.

### Update a VPS

1. Operator opens the VPS list.
2. Operator selects `Edit` for a row.
3. Operator changes metadata or connection settings.
4. System validates and saves the changes.
5. System updates the `updatedAt` and `updatedBy` fields.

### Investigate a communication problem

1. Operator opens the VPS list.
2. Operator sees a VPS with `offline` or `degraded` status.
3. Operator opens the logs view.
4. System shows recent requests, responses, failures, and timestamps.
5. Operator uses correlation and error data to diagnose the issue.

## Acceptance Criteria

The subsystem is considered complete when:

1. Operators can create, edit, and delete VPS records from the Admin Dashboard.
2. The main application screen uses a left vertical menu and a right working area with a top title section.
3. The `Remote VPS` list displays required VPS information and actions.
4. Every Control Center to Remote Controller interaction is logged with VPS linkage, timestamp, result, and enough metadata for troubleshooting.
5. Interaction logs are searchable by VPS and time range.
6. Sensitive data is not stored in plaintext in registry records or logs.
